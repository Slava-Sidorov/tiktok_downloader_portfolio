# Roadmap: Модернизация → универсальный загрузчик TikTok / YouTube

> **Источники истины:** [TZ_Modernization_v1.md](TZ_Modernization_v1.md) (что делаем) +
> [TechnicalBuilding_v3.md](TechnicalBuilding_v3.md) (текущая реализация).
> **Правило:** каждая фаза — отдельная ветка/PR, заканчивается зелёными
> `gofmt -l`, `go vet ./...`, `go build ./...`, `go test ./...`.
> Регресс TikTok-через-прокси проверяется в конце **каждой** фазы (не должен меняться).

## Принципы (не пересматриваются)

- База — текущий Go-проект; боевая логика (пул прокси, замер скорости, авто-воркеры,
  блэклист, graceful shutdown, маскировка кредов, защита от argument injection) сохраняется.
- Чистый флаговый CLI, без wizard.
- yt-dlp — subprocess, как сейчас.
- Обратная совместимость: запуск без новых флагов работает как раньше. Единственное
  осознанное отличие — дефолтная пауза становится рандомной `200–800ms` (см. Фаза 1).

## Фаза 0 — Подготовка: конфиг вместо позиционных аргументов

**Зачем:** `runYtDlp(ctx, proxy, url, outDir)` (main.go:647) и `worker(ctx, id, urls,
proxies, idx, delay, outDir, archive, wg)` (main.go:683) в фазах 1/3/5 обрастают
параметрами: опциональный прокси, формат аудио/видео, bypass-бандл, шаблон имени,
`--limit-rate`, диапазон джиттера, режим no-proxy. Вводим структуры один раз.

**Задачи:**
- Ввести `type runConfig struct` с полями будущих фич (audio bool, bypass bool,
  limitRate string, nameTemplate string, noProxy bool, delayMin/delayMax time.Duration,
  platform *platformProfile — заполняется в Фазе 2). Пока заполняется дефолтами,
  повторяющими текущее поведение.
- Переписать `runYtDlp` на приём `cfg` (или `ytdlpArgs(cfg, proxy, url, outDir)` +
  тонкий раннер) — **поведение байт-в-байт как сейчас**.
- Переписать `worker` на приём `cfg` вместо россыпи аргументов; обновить точку вызова
  (main.go:992).

**Тесты / приёмка:**
- `go build`, `go test` зелёные; регресс TikTok-через-прокси не изменился.
- (Опц.) unit-тест `ytdlpArgs`: для дефолтного cfg собирает ровно текущий набор флагов.

**Definition of Done:** диффа поведения ноль, сигнатуры готовы принимать новые фичи.

## Фаза 1 — Джиттер задержки + режим `--no-proxy`

**Соответствие ТЗ:** §3.2, §3.3. Ядро изменений в `worker`.

**Задачи:**
- Флаги: `--delay-min`, `--delay-max`, `--no-proxy`, `--limit-rate`.
  `--delay` оставить как legacy-шорткат (`min = max = X`), пометить в help.
- Дефолты джиттера: прокси-режим `200ms–800ms`, no-proxy `2s–4s`.
- Джиттер: заменить фиксированный `delay` (main.go:761) на
  `delayMin + rand*(delayMax-delayMin)` на каждой итерации.
- Ветка `--no-proxy` в `worker`:
  - строго 1 воркер; прямое соединение (в `runYtDlp` `--proxy` не добавляется);
  - без `nextProxy` / блэклиста / `incFailAndMaybeBan`;
  - retryable-ошибки → до `maxAttempts` (3) по той же ссылке с паузой-джиттером
    **между ретраями** (не только между ссылками);
  - `--limit-rate` default `5M` (отключается `--limit-rate 0`); в прокси-режиме выкл.
- В `main()`: пропустить замер скорости и авто-расчёт воркеров при `--no-proxy`.
- **Валидация конфликтов:**
  - `--no-proxy` + `--workers N>1` → ошибка конфигурации (явный выход, не тихий clamp);
  - `--no-proxy` + явный `--proxies` → предупреждение, файл игнорируется;
  - `0 ≤ delay-min ≤ delay-max`, иначе ошибка.

**Тесты:**
- джиттер в границах `[min, max]` (много прогонов);
- валидация `min ≤ max`;
- валидация `--no-proxy` + `--workers 3` → ошибка.

**Definition of Done:** оба сетевых режима работают; TikTok-через-прокси = регресс ок;
`--no-proxy` качает в 1 поток с паузами и ретраями.

## Фаза 2 — Платформенный профиль (`--platform tiktok|youtube`)

**Соответствие ТЗ:** §3.1. Дефолт `tiktok` (полная обратная совместимость).

**Задачи:**
- `type platformProfile` c методами: `extractID(url) string`, `dedupKey(url) string`,
  `videoMarkers []marker`. Two instances: `tiktokProfile`, `youtubeProfile`.
- Обобщить `extractTikTokID` (main.go:534) и `dedupKey` (main.go:542) за профиль.
  TikTok — как сейчас (короткие `vm.tiktok.com` без ID → дедуп по URL).
  YouTube — ID из `watch?v=`, `youtu.be/`, `shorts/`, `embed/`.
- Флаг `--platform` (default `tiktok`), проброс профиля в `runConfig`.
- `classifyYtdlpError` (main.go:606): добавить YouTube video-маркеры
  («Video unavailable», «Private video», «This video is not available»,
  «Sign in to confirm…»). **403/429 остаются сетевыми (retryable)** — не в video-маркеры.
- Pre-check по `.archive.txt`: формат yt-dlp `extractor id` уже различает платформы,
  один файл архива обслуживает обе — проверить, что pre-check в `worker` (main.go:696)
  использует профильный `extractID`.

**Тесты:**
- извлечение ID обеих платформ (все формы URL);
- дедуп: разные формы одной ссылки → одна запись;
- классификация YouTube video-ошибок vs 403/429 (сетевые).

**Definition of Done:** `--platform youtube` извлекает ID/дедупит/классифицирует
корректно; TikTok не изменился.

## Фаза 3 — Аудио-режим `--audio`

**Соответствие ТЗ:** §3.6. Главный YouTube-кейс (музыка, подкасты).

**Задачи:**
- Флаг `--audio`.
- Параметризовать формат в `runYtDlp` (main.go:651):
  - видео (default): `-f bestvideo*+bestaudio/best --merge-output-format mp4`,
    имя `%(id)s.%(ext)s` — без изменений;
  - аудио: `-f bestaudio/best -x`, **без перекодирования** (никакого `--audio-format`),
    имя `%(title)s.%(ext)s` + `--windows-filenames`.
- Доступен на обеих платформах (запрещать TikTok не нужно).

**Тесты:**
- (unit) `ytdlpArgs`: `--audio` даёт аудио-набор флагов + title-шаблон;
  без него — прежний видео-набор.

**Definition of Done:** `--platform youtube --audio` качает аудио, имена по title;
видео-режим байт-в-байт как раньше.

## Фаза 4 — Сабкоманда `parse` (препроцессинг сырого текста)

**Соответствие ТЗ:** §3.4. Только препроцессинг, без сети.

**Задачи:**
- `downloader parse --input raw.txt [--platform tiktok|youtube] [--output links.txt]`.
- Regex-поиск URL в неструктурированном тексте; фильтр по доменам платформы
  (`youtube.com`/`youtu.be` либо `tiktok.com`/`vm.tiktok.com`).
- Очистка:
  - хвостовая пунктуация `.,!?;:)»"`;
  - мусорные GET-параметры (`list`, `index`, `t`, `si`, `pp`…),
    **`v` у watch-ссылок сохраняется** — это ID;
  - короткие `vm.tiktok.com` — как есть, без резолва.
- Дедуп по каноническому ID (профиль из Фазы 2), иначе по нормализованному URL.
- Выход: перезапись `--output` (default `links.txt`); статистика:
  строк обработано / URL найдено / уникальных записано.

**Тесты (ключевой пункт приёмки ТЗ §6):**
- грязный текст → чистый список: watch с `&list=&index=&t=`, `youtu.be`, `shorts`,
  TikTok полная и `vm.tiktok.com`, дубли в разных формах → одна запись.

**Definition of Done:** `parse` из сырого текста даёт валидный `links.txt`,
сохраняет `v`, режет мусор, дедупит; сеть не трогается.

## Фаза 5 — Флаг `--yt-bypass` (обход 403/429, только YouTube)

**Соответствие ТЗ:** §3.5. Один флаг = бандл.

**Задачи:**
- Флаг `--yt-bypass`. При установке добавляет к вызову yt-dlp:
  `--extractor-args "youtube:player_client=android,ios"` + `--force-ipv4`.
- Только `--platform youtube`; при `tiktok` — предупреждение и игнор.
- Действует на весь прогон, без скрытых фаз/автоэскалации.

**Тесты:**
- (unit) `ytdlpArgs`: с `--yt-bypass` + youtube добавляет оба флага;
  с tiktok — не добавляет (и предупреждает).

**Definition of Done:** флаги доходят до yt-dlp на YouTube; на TikTok игнор с варнингом.
Сценарий добивания `failed.txt` работает (см. приёмка §6.3).

       (unit + реальный прогон: yt-dlp принял флаги без ошибок опций).
4. [x] `parse` сырого текста → валидный `links.txt` (реальный прогон: `v`
       сохранён, мусор срезан, дубли схлопнуты, статистика верна).
