# ТЗ: Модернизация tiktok-downloader → универсальный загрузчик TikTok / YouTube

## 1. Цель и контекст

Расширить существующую Go-утилиту `tiktok-downloader` до универсального CLI-загрузчика для **TikTok и YouTube**, поглотив функциональность Python-скриптов (`download_tracks`, `parse_yt_links`) и требований из ТЗ YouTube-проекта.

**Принципиальные решения (зафиксированы, не пересматриваются):**

- База — текущий Go-проект. Вся боевая логика сохраняется как есть: пул прокси с замером скорости, авторасчёт воркеров, блэклист с мягким баном, graceful shutdown, маскировка кредов, защита от argument injection.
- UX — **чистый флаговый CLI**, без интерактивного wizard.
- yt-dlp вызывается как subprocess (как сейчас), Python API не используется.

## 2. Явно вне scope (отклонено при обсуждении)

| Что | Почему |
|---|---|
| Интерактивный wizard | выбран флаговый CLI |
| Парсинг title страниц («Название + Ссылка») | нишевая личная фича, к закачке не относится |
| Отдельный режим/сабкоманда докачки | уже покрыто: повторный запуск с `--links failed.txt` + `.archive.txt` |
| Автопрогон упавших ссылок с bypass-флагами внутри одного запуска | перегруз: требует ре-классификации ошибок и создаёт скрытую вторую фазу; эскалацию делает пользователь вручную (см. 3.5) |
| Любая постобработка / перекодирование скачанного | инструмент только качает в лучшем качестве; что делать со скачанным дальше — решает пользователь |
| Переименование бинаря/проекта (`tiktok-downloader` → нейтральное имя) | по желанию, отдельно от этого ТЗ |

## 3. Новая функциональность

### 3.1 Профиль платформы: `--platform tiktok|youtube`

Значение по умолчанию — `tiktok` (полная обратная совместимость).

Профиль инкапсулирует всё платформо-специфичное:

- **Извлечение канонического ID** (для pre-check по архиву и дедупа):
  - TikTok — как сейчас (`extractTikTokID`; короткие `vm.tiktok.com` без ID — дедуп по URL);
  - YouTube — ID из форм `watch?v=`, `youtu.be/`, `shorts/`, `embed/`.
- **Маркеры video-ошибок** для `classifyYtdlpError`: дополнить YouTube-маркерами («Video unavailable», «Private video», «This video is not available», «Sign in to confirm...» и т.п.). 403/429 классифицируются как сетевые (retryable).
- **Доступность `--yt-bypass`** (см. 3.5).

Оба профиля поддерживают оба сетевых режима — и через прокси, и без (`--no-proxy`).

Технически: `extractTikTokID` / `dedupKey` обобщаются за интерфейс/структуру профиля; pre-check по `.archive.txt` учитывает префикс экстрактора (формат yt-dlp `extractor id` уже различает платформы, один файл архива работает для обеих).

### 3.2 Режим без прокси: `--no-proxy`

Режим «локальный IP, бронебойный, медленный» — портирован из `download_tracks/main.py` и раздела 4.3 ТЗ YouTube.

- Прямое соединение, **строго 1 воркер**.
- Замер скорости прокси и авторасчёт воркеров пропускаются полностью.
- Блэклист/бан-логика прокси отключена. Retryable-ошибки → до `maxAttempts` (3) попыток по той же ссылке с паузой-джиттером между попытками.
- Дефолтный джиттер задержки — **2–4 с** (имитация человека, см. 3.3).
- Мягкий лимит скорости: passthrough-флаг `--limit-rate` в yt-dlp; в `--no-proxy` режиме default `5M` (отключается `--limit-rate 0`), в прокси-режиме по умолчанию выключен.

**Валидация конфликтов:**

- `--no-proxy` + `--workers N>1` → ошибка конфигурации (явный отказ, не тихий clamp);
- `--no-proxy` + явно указанный `--proxies` → предупреждение, файл прокси игнорируется.

### 3.3 Джиттер задержки: `--delay-min` / `--delay-max`

Фиксированная пауза заменяется диапазоном с равномерным рандомом на каждую итерацию воркера (перенос `random.uniform(...)` из всех трёх Python-скриптов).

| Режим | Дефолт | Обоснование |
|---|---|---|
| Через прокси | `200ms–800ms` | каждый запрос с другого IP, но разброс человечнее фикса |
| `--no-proxy` | `2s–4s` | единственный IP, имитация человека (как в `download_tracks/main.py:64`) |

- Валидация: `0 ≤ min ≤ max`.
- Обратная совместимость: существующий `--delay X` остаётся как шорткат `min = max = X` (пометить в help как legacy).
- В `--no-proxy` режиме джиттер применяется также между ретраями одной ссылки.

### 3.4 Сабкоманда `parse` — препроцессинг сырого текста

Портирует `parse_yt_links/extr_links.py` + раздел 4.1 ТЗ YouTube.

```
downloader parse --input raw.txt [--platform tiktok|youtube] [--output links.txt]
```

- Вход — файл с **любым неструктурированным текстом** (история просмотров, скопированная страница).
- Regex-поиск URL, фильтрация по доменам выбранной платформы (`youtube.com`/`youtu.be` либо `tiktok.com`/`vm.tiktok.com`).
- Очистка:
  - хвостовая пунктуация (`.,!?;:)»"` — как `extr_links.py:23`);
  - мусорные GET-параметры: `list`, `index`, `t`, `si`, `pp` и т.п. **Критично: параметр `v` у watch-ссылок сохраняется** — это сам ID.
  - короткие `vm.tiktok.com` ссылки сохраняются как есть (без резолва).
- Дедупликация по каноническому ID (если извлекается), иначе по нормализованному URL.
- Выход: перезапись `--output` (default `links.txt`); статистика в консоль: строк обработано / URL найдено / уникальных записано.
- Только препроцессинг: сетевых запросов сабкоманда не делает.

### 3.5 Флаг `--yt-bypass` — обход 403/429 (только YouTube)

Один флаг, добавляющий к вызову yt-dlp **бандл** из двух параметров (источник — `sec_kz_vpn.py:49` и `download_tracks/main.py:37`):

```
--extractor-args "youtube:player_client=android,ios"
--force-ipv4
```

- Работает только при `--platform youtube`; при `tiktok` — предупреждение и игнор (у TikTok таких рычагов нет).
- Действует на весь прогон (без скрытых фаз и автоэскалации).
- **Рекомендуемый сценарий использования — задокументировать в README:**

```bash
# обычный прогон
downloader --platform youtube --no-proxy --links links.txt
# добить упавшее на 403/429:
downloader --platform youtube --no-proxy --links downloads/failed.txt --yt-bypass
```

- Если позже понадобится `--force-ipv4` отдельно (кривой IPv6 на первом прогоне) — расщепить на самостоятельный флаг; сейчас не делать.

### 3.6 Аудио-режим: `--audio`

**Почему пункт появился:** главный сценарий YouTube-скриптов — скачивание **аудио** (музыка, подкасты): `download_tracks` качает `bestaudio` + извлекает аудио. Текущий `runYtDlp` жёстко зашит под видео (`-f bestvideo*+bestaudio/best --merge-output-format mp4`, main.go:651), поэтому без переключателя аудио/видео объединённый инструмент не покрывает основной YouTube-кейс.

- `--audio`: заменяет формат на `-f bestaudio/best` + `-x` (извлечение аудио-дорожки без видео).
- **Без перекодирования.** Аудио сохраняется в исходном лучшем контейнере, который отдаёт платформа (например m4a / webm). Никакого форсирования кодека, никакого выбора `--audio-format`. Цель — лучшее качество; что делать со скачанным дальше, решает пользователь.
- Именование файлов: в аудио-режиме default-шаблон `%(title)s.%(ext)s` + `--windows-filenames` (для музыки/подкастов имена по ID бесполезны); видео-режим не меняется (`%(id)s.%(ext)s`).
- Ориентирован на YouTube (подкасты, музыка). Технически доступен и на TikTok — специально запрещать незачем, но кейс экзотический.

## 5. Обратная совместимость

- Запуск без новых флагов работает как раньше, **одно осознанное отличие**: пауза по умолчанию становится рандомной `200–800ms` вместо фиксированных `500ms`. Явный `--delay X` возвращает старое фиксированное поведение.
- `.archive.txt` совместим — формат yt-dlp уже содержит имя экстрактора.
- `failed.txt` / `failed_reasons.txt` / счётчики — без изменений.

## 6. Тесты и приёмка

**Unit-тесты** (в дополнение к `proxy_parse_test.go`):

- `parse`: грязный текст → чистый список (watch-ссылка с `&list=&index=&t=`, `youtu.be`, `shorts`, TikTok полная и `vm.tiktok.com`, дубли в разных формах → одна запись);
- извлечение ID обеих платформ;
- джиттер в границах `[min, max]`, валидация `min ≤ max`;
- валидация конфликтов флагов (`--no-proxy` + `--workers 3` → ошибка).

**Ручная приёмка:**

1. TikTok через прокси — поведение идентично текущему (регресс).
2. `--platform youtube --no-proxy --audio` — треки скачиваются с паузами 2–4 с, имена по title.
3. Повторный прогон `failed.txt` с `--yt-bypass` — флаги доходят до yt-dlp, 403-ссылки добиваются.
4. `parse` сырого текста → валидный `links.txt`.
5. Недоступные видео не роняют процесс; счётчики и `failed.txt` корректны.

## 7. Предлагаемый порядок работ

4. Сабкоманда `parse`;
5. `--yt-bypass`;
6. README + тесты (тесты — по ходу каждого пункта).
