# TikTok / YouTube Downloader

CLI-утилита на Go для массового скачивания видео и аудио из **TikTok и YouTube** через `yt-dlp` — с пулом прокси или напрямую (`--no-proxy`).

## Требования

| Инструмент | Установка |
|---|---|
| `yt-dlp` | `pip install yt-dlp` |
| `ffmpeg` | https://ffmpeg.org/download.html |

Оба должны быть доступны в `PATH`. При отсутствии любого из них утилита завершится с понятным сообщением до начала загрузки.

## Установка

```bash
go build -o tiktok-downloader .
```

## Использование

```bash
tiktok-downloader [флаги]                 # массовое скачивание
tiktok-downloader parse --input raw.txt   # извлечь ссылки из сырого текста → links.txt
tiktok-downloader update                  # обновить yt-dlp до последней версии
tiktok-downloader check-proxies [флаги]   # диагностика прокси, без скачивания
tiktok-downloader test-host <url>         # добавить хост для замера скорости навсегда
```

### Флаги основного режима

| Флаг | По умолчанию | Описание |
|---|---|---|
| `--links` | `links.txt` | Путь к файлу со ссылками (одна ссылка на строку) |
| `--platform` | `tiktok` | Платформа: `tiktok` \| `youtube` |
| `--proxies` | `proxies.txt` | Путь к файлу с прокси (в `--no-proxy` игнорируется) |
| `--output` | `./downloads` | Папка для сохранения |
| `--workers` | `0` | Кол-во воркеров (0 = автоопределение; в `--no-proxy` всегда 1) |
| `--no-proxy` | `false` | Прямое соединение без прокси: 1 воркер, паузы 2–4 с, лимит 5M |
| `--audio` | `false` | Скачивать только аудио (лучшее качество, без перекодирования); имена по title |
| `--yt-bypass` | `false` | Обход 403/429 (только YouTube): mobile-клиент + force-ipv4 |
| `--delay-min` | `200ms` / `2s`* | Нижняя граница случайной паузы между запросами воркера |
| `--delay-max` | `800ms` / `4s`* | Верхняя граница случайной паузы |
| `--delay` | — | **[legacy]** фиксированная пауза; шорткат `delay-min = delay-max = X` |
| `--limit-rate` | выкл / `5M`* | Лимит скорости yt-dlp (`5M`, `500K`); `0` — снять лимит |
| `--socks5` | `false` | Использовать socks5 вместо http для прокси без явной схемы |
| `--log` | _(не задан)_ | Путь к файлу лога; без флага — только консоль |
| `--verbose` | `false` | Подробный вывод yt-dlp для каждого видео |
| `--size` | `15.0` | Примерный размер видео в МБ (влияет на расчёт числа воркеров) |
| `--test-url` | _(не задан)_ | Свой хост для замера скорости прокси на один запуск (пробуется первым) |

\* — второе значение действует в режиме `--no-proxy`.

### Примеры

```bash
# TikTok через прокси (как раньше)
tiktok-downloader --links links.txt --proxies proxies.txt --output ./downloads --log run.log

# YouTube: аудио (музыка/подкасты) без прокси, паузы 2–4 с, имена по названию
tiktok-downloader --platform youtube --no-proxy --audio --links links.txt

# YouTube: видео через прокси
tiktok-downloader --platform youtube --links links.txt --proxies proxies.txt
```

## Режим без прокси (`--no-proxy`)

«Локальный IP, бронебойный, медленный» — для аккуратного скачивания без пула прокси:

- прямое соединение, **строго 1 воркер** (`--workers N>1` — ошибка конфигурации);
- замер скорости и авто-расчёт воркеров пропускаются; файл прокси не нужен;
- случайные паузы **2–4 с** между запросами и между повторами одной ссылки (имитация человека);
- мягкий лимит скорости **5M** по умолчанию (`--limit-rate 0` — снять);
- сетевые ошибки → до 3 попыток по той же ссылке;
- TLS-проверка сертификатов **включена** (отключается только в прокси-режиме).

## Паузы (джиттер)

Пауза между запросами воркера — случайная из диапазона `[--delay-min, --delay-max]`:
через прокси по умолчанию `200–800 мс`, в `--no-proxy` — `2–4 с`. Старый флаг
`--delay X` работает как фиксированная пауза (`min = max = X`).

## Докачка упавших ссылок (YouTube 403/429)

403/429 считаются сетевыми ошибками: ссылка попадает в `failed.txt`. Рекомендуемый сценарий — добить их отдельным прогоном с `--yt-bypass`:

```bash
# обычный прогон
tiktok-downloader --platform youtube --no-proxy --links links.txt
# добить упавшее на 403/429:
tiktok-downloader --platform youtube --no-proxy --links downloads/failed.txt --yt-bypass
```

`--yt-bypass` добавляет к вызову yt-dlp `--extractor-args "youtube:player_client=android,ios"` и `--force-ipv4`. Работает только с `--platform youtube` (на TikTok — предупреждение и игнор) и действует на весь прогон.

## Аудио-режим (`--audio`)

- Скачивается только звуковая дорожка в **лучшем исходном качестве без перекодирования** — контейнер тот, что отдаёт платформа (`m4a`, `webm`/`opus` и т.п.).
- Имена файлов — по названию (`%(title)s.%(ext)s` + `--windows-filenames`); в видео-режиме — по ID, как раньше.
- Основной кейс — YouTube (музыка, подкасты); технически работает и на TikTok.

## Форматы прокси

Файл `proxies.txt` — по одному прокси на строку. Пустые строки и строки начинающиеся с `#` игнорируются. Невалидные строки пропускаются с предупреждением (креды в предупреждении маскируются), программа не падает.

Поддерживаются четыре формата:

| Формат | Пример |
|---|---|
| Готовый URL (pass-through) | `http://alice:secret@1.2.3.4:8080` |
| `USER:PASS@IP:PORT` | `alice:secret@1.2.3.4:8080` |
| `IP:PORT:USER:PASS` | `1.2.3.4:8080:alice:secret` |
| `IP:PORT` | `1.2.3.4:8080` |

Протокол для форматов без схемы определяется флагом `--socks5` (по умолчанию `http`).

### Ограничения форматов прокси

- **Пароль с `:`** — формат `IP:PORT:USER:PASS` не поддерживает двоеточие в пароле (строка не разберётся корректно). Используйте формат с `@` (`USER:PASS@IP:PORT`) или полный URL (`http://user:p:ass@host:port`).
- **IPv6** — голый IPv6-адрес не поддерживается. Используйте только полную схему с квадратными скобками: `http://[2001:db8::1]:8080`.

## Автоопределение числа воркеров и замер скорости

Если `--workers 0` (по умолчанию), перед скачиванием утилита замеряет скорость прокси и рассчитывает число воркеров:

1. Пул перемешивается, берётся выборка **`max(ceil(10% пула), 5)`**, но не больше размера пула. Практически: пул > 50 → ~10%; пул 5–50 → 5 штук; пул < 5 → весь пул.
2. Выборка тестируется параллельно (до 5 одновременно); при наборе **10 рабочих** замер останавливается досрочно.
3. По средней скорости рабочих прокси считается число воркеров (в пределах **1–20**).
4. Если из выборки **не заработал ни один** прокси — программа завершается с понятным сообщением.

При `--workers N` (N > 0) замер пропускается — используется указанное число воркеров.

### Тестовые хосты для замера

Скорость замеряется скачиванием ~1 МБ файла. Хосты пробуются по порядку, первый ответивший даёт замер — падение одного хоста не роняет авто-режим. Порядок приоритета:

1. `--test-url <url>` — разовый хост на текущий запуск (первым в очереди);
2. постоянные хосты, добавленные командой `test-host` (см. ниже);
3. встроенные fallback-и: `proof.ovh.net`, `speed.cloudflare.com`, `speedtest.tele2.net`.

Дубликаты в списке убираются.

## Сабкоманды

### `update`

```bash
tiktok-downloader update
```

Запускает `yt-dlp -U` и транслирует его вывод.

### `check-proxies` — диагностика прокси

```bash
tiktok-downloader check-proxies [--proxies proxies.txt] [--socks5] [--test-url URL] [--all]
```

Замеряет скорость прокси и печатает таблицу: живые сверху по убыванию скорости, мёртвые снизу с причиной. **Ничего не качает и не банит.** Креды прокси в выводе маскируются.

- по умолчанию — та же выборка, что и в авто-режиме (~10% пула, до 10 рабочих);
- `--all` — проверить **весь** пул;
- итоговая строка показывает охват (`проверено N из M` или `весь пул M`) и среднюю скорость живых.

### `test-host` — постоянный хост для замера

```bash
tiktok-downloader test-host https://мой.хост/1mb.bin
```

Добавляет хост в **постоянный** список для замера скорости прокси и сразу выходит (загрузку не запускает). Полезно, когда встроенные хосты недоступны в сети клиента: одна команда навсегда прописывает рабочий хост.

- список хранится в `<user-config-dir>/tiktok-downloader/test-hosts.txt` (на Windows — `%AppData%`, на Linux — `~/.config`, на macOS — `~/Library/Application Support`);
- принимается только полная `http`/`https` ссылка; дубликаты не добавляются;
- добавленные хосты пробуются раньше встроенных.

### `parse` — извлечение ссылок из сырого текста

```bash
tiktok-downloader parse --input raw.txt [--platform tiktok|youtube] [--output links.txt]
```

Превращает **любой неструктурированный текст** (история просмотров, скопированная страница) в чистый список ссылок для скачивания. Без сетевых запросов.

- находит URL и оставляет только ссылки выбранной платформы (`youtube.com`/`youtu.be` либо `*.tiktok.com`);
- срезает прилипшую пунктуацию (`.,!?;:)»"` и т.п.) и мусорные GET-параметры (`list`, `index`, `t`, `si`, …) — **параметр `v` у watch-ссылок сохраняется**, это сам ID;
- короткие `vm.tiktok.com` сохраняются как есть (без резолва);
- дубли одной ссылки в разных формах (`watch?v=`, `youtu.be/`, `shorts/`) схлопываются в одну запись;
- результат **перезаписывает** `--output` (default `links.txt`); в консоль — статистика: строк обработано / URL найдено / уникальных записано.

## Поведение при ошибках

### Блэклист прокси (мягкий бан)

Прокси не банится сразу при первой ошибке. Бан наступает после **2 фейлов подряд** (глобально, от любого воркера). Успешная загрузка обнуляет счётчик фейлов для прокси.

Ошибки делятся на два класса:

- **VIDEO-ошибка** (видео удалено, приватное, geo-блок, 404, «Sign in to confirm» и т.п.) — повтор не поможет, ссылка сразу записывается в `failed` с причиной, дальнейшие попытки не делаются. Наборы маркеров свои у каждой платформы (`--platform`).
- **Сетевая ошибка** (таймаут, отказ соединения, 407, **403/429** и т.п.) — ссылка ретраится; в прокси-режиме счётчик фейлов прокси увеличивается, при пороге прокси попадает в блэклист. В `--no-proxy` бан-логики нет — до 3 повторов той же ссылки с паузой.

Для YouTube массовые 403/429 добиваются отдельным прогоном `failed.txt` с `--yt-bypass` (см. выше).

> Так как бан наступает только после 2 фейлов **подряд по одному прокси**, а `nextProxy` раздаёт прокси по кругу, три попытки одной ссылки не гарантированно идут по трём разным прокси — на маленьком пуле возможен повтор. Бан в основном накапливается по разным ссылкам.

### Каждая ссылка получает максимум 3 попытки на разных (по возможности) прокси.

### Файлы с результатами

После завершения в папке `--output` создаются:

- `failed.txt` — чистый список ссылок для повторного прогона.
- `failed_reasons.txt` — ссылки с причиной фейла (tab-separated).
- `.archive.txt` — скрытый архив yt-dlp; ссылки из него пропускаются при повторном запуске без сетевых запросов.

### Счётчики

Итог выводится тремя счётчиками: **скачано** / **пропущено** (уже было в архиве) / **ошибок** (после всех попыток). Короткие ссылки (`vm.tiktok.com`), которые yt-dlp определил как уже скачанные по реальному ID, тоже засчитываются как пропущенные.

## Graceful shutdown

`Ctrl+C` (или `SIGTERM`) мягко останавливает работу: запущенные процессы `yt-dlp` убиваются через контекст, новые загрузки не стартуют, статистика и `failed.txt` записываются. **Повторный `Ctrl+C`** форсирует немедленный выход (код 130), если какой-то `yt-dlp` завис на закрытии соединения.

## Безопасность

### Ссылки как аргументы yt-dlp

Ссылки из `links.txt` передаются yt-dlp после разделителя `--`, поэтому строка, начинающаяся с `-`, не может быть истолкована как опция yt-dlp (защита от argument injection из недоверенного файла ссылок).

### Маскировка кредов прокси

Логин/пароль прокси маскируются (`//***@`) в консоли, в файле `--log` и в диагностике `check-proxies` — креды не оседают в логах, которые можно переслать для разбора.

## Known limitations

### TLS-проверка отключена в прокси-режиме (`--no-check-certificates`)

При работе **через прокси** `yt-dlp` запускается с `--no-check-certificates` — для совместимости с прокси, у которых нестандартные TLS-цепочки. **Не скачивайте чувствительный контент через недоверенные прокси** — возможен MITM. В режиме `--no-proxy` проверка сертификатов включена как обычно.

### Гонка `.archive.txt` на сетевых ФС

Несколько воркеров параллельно пишут в один файл `.archive.txt`. На локальном диске это работает корректно. На сетевых файловых системах (NFS, SMB, сетевые диски) возможна гонка записей — **используйте локальный диск** для папки `--output`.
